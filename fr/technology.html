SocialFlow

Les choix techniques derrière SocialFlow

Pas d'extension Chrome, pas de version web, pas de Flutter ni d'Expo. La transparence technique fait partie de nos valeurs — voici les vraies raisons.

Ce que SocialFlow fait de fondamentalement différent

La promesse de SocialFlow est simple : vous connecter simultanément à plusieurs comptes sur chaque réseau social, dans des sessions complètement isolées les unes des autres. Profil perso, profil pro, profil client — chacun avec ses propres connexions, ses propres cookies, ses propres sessions.

Ce n'est pas juste un dashboard qui affiche vos réseaux dans des onglets. C'est une isolation réelle : quand vous êtes sur votre profil "Pro" et que vous ouvrez Facebook, c'est votre compte Facebook professionnel qui apparaît. Quand vous basculez sur le profil "Perso", c'est votre compte personnel, sans avoir à vous déconnecter et reconnecter.

Cette isolation repose sur un mécanisme technique précis : chaque profil dispose de son propre espace de stockage de cookies, totalement séparé des autres. C'est ce qu'on appelle un "cookie jar" isolé.

Et c'est exactement là que les choses se compliquent avec Chrome et le web.

Pourquoi pas d'extension Chrome ?

Le problème fondamental : Chrome partage les cookies entre tous les onglets

Dans Chrome, tous les onglets partagent le même espace de cookies. Si vous vous connectez à Facebook dans un onglet, vous êtes connecté au même compte Facebook dans tous les autres onglets. C'est un choix de conception de Google, et aucune extension ne peut changer ce comportement.

Une extension Chrome peut lire et modifier des cookies individuels via l'API officielle de Chrome. Mais elle ne peut pas créer des espaces de cookies séparés par onglet ou par profil. Le cookie que vous modifiez dans un onglet est immédiatement visible dans tous les autres.

L'approche "interception réseau" : théoriquement possible, pratiquement fragile

Il existe une approche théorique : intercepter chaque requête réseau sortante pour remplacer les cookies envoyés, et intercepter chaque réponse pour capturer les cookies reçus et les stocker dans un espace virtuel géré par l'extension. C'est ce que font certaines extensions payantes comme SessionBox.

Le problème, c'est que Google a profondément modifié les règles du jeu avec Manifest V3, la nouvelle norme pour les extensions Chrome :

  • Les extensions ne peuvent plus modifier les requêtes réseau en temps réel. L'ancienne API qui le permettait a été remplacée par un système de règles déclaratives, beaucoup moins flexible.
  • Les règles déclaratives ne sont pas conçues pour du cookie-swapping dynamique. Elles fonctionnent avec des règles statiques ou semi-statiques, pas avec de la logique qui change à chaque requête en fonction de l'onglet actif.
  • Les cookies définis par les sites dans les réponses HTTP arrivent quand même dans l'espace partagé du navigateur, ce qui crée des conflits constants entre les profils.

Le résultat : même les extensions qui tentent cette approche rencontrent des déconnexions aléatoires, des sessions qui se mélangent, et des incompatibilités avec les sites qui utilisent des mécanismes de sécurité avancés (ce que font tous les réseaux sociaux majeurs).

Et Firefox ?

Firefox est le seul navigateur qui propose un vrai mécanisme d'isolation : les Containers (Multi-Account Containers). Chaque container a son propre espace de cookies, complètement séparé. Et les extensions Firefox ont accès à une API officielle pour créer et gérer ces containers.

C'est exactement ce dont SocialFlow aurait besoin. Mais cette API n'existe que sur Firefox — Chrome n'a jamais implémenté d'équivalent, et Google n'a montré aucune intention de le faire.

Nous pourrions proposer une extension Firefox avec les Containers, mais cela ne couvrirait qu'une fraction de nos utilisateurs. Nous avons préféré concentrer nos efforts sur une solution qui fonctionne partout.

Ce qu'une extension Chrome pourrait quand même faire

Une extension Chrome pourrait offrir certaines fonctionnalités de SocialFlow : la sidebar avec tous les réseaux, le mode niveaux de gris pour réduire les distractions, l'injection de scripts personnalisés dans les pages. Mais sans l'isolation des profils, ce serait SocialFlow amputé de sa fonctionnalité principale. Nous avons estimé que proposer un produit incomplet ne rendrait service à personne.

Pourquoi pas de version web ?

Les navigateurs web sont conçus pour empêcher exactement ce que SocialFlow fait

Sur le web, un site ne peut pas afficher un autre site à l'intérieur de lui-même. C'est une règle de sécurité fondamentale du web, et elle existe pour de bonnes raisons : elle empêche un site malveillant de se faire passer pour votre banque ou votre réseau social en l'embarquant dans une fausse interface.

Concrètement, quasiment tous les réseaux sociaux (Facebook, Twitter, Instagram, LinkedIn, etc.) envoient un en-tête HTTP qui interdit à tout autre site de les afficher dans un cadre intégré. C'est une protection standard contre les attaques de type "clickjacking".

Même si un réseau social n'avait pas cette protection, le navigateur empêcherait le site SocialFlow d'accéder aux cookies de Facebook ou Twitter. C'est le principe de la "Same-Origin Policy" : chaque site est isolé des autres. Ironiquement, c'est exactement le type d'isolation que nous voudrions contrôler — mais elle est imposée par le navigateur, pas par nous.

Une version web serait un simple tableau de liens

Si nous proposions une version web, la seule chose qu'elle pourrait faire serait d'afficher des liens vers les réseaux sociaux, qui s'ouvriraient dans de nouveaux onglets. Pas d'isolation de profil, pas d'interface unifiée, pas d'injection de scripts. Autant mettre des favoris dans votre navigateur.

C'est pour cette raison que ce site est une page de présentation et non l'application elle-même.

Pourquoi l'application native (Tauri) résout tout ça

SocialFlow est construit avec Tauri, un framework qui crée des applications natives à partir de technologies web. Voici ce que ça change :

  • Chaque réseau social s'affiche dans un composant WebView natif, directement intégré dans l'application. Ce n'est pas un onglet de navigateur : c'est un mini-navigateur contrôlé par l'application, avec son propre espace de cookies.
  • L'application contrôle les cookies de chaque WebView. Quand vous changez de profil, SocialFlow sauvegarde les cookies du profil actuel et charge ceux du nouveau profil. Chaque réseau, dans chaque profil, a ses propres cookies.
  • L'application peut injecter des scripts dans les WebViews dès le tout premier instant du chargement de la page (avant même que le site ne commence à s'afficher). C'est ce qui permet le mode niveaux de gris, le filtre "amis uniquement", et d'autres personnalisations.
  • Aucune restriction d'affichage. Les en-têtes de sécurité qui empêchent l'intégration dans un site web ne s'appliquent pas aux WebViews natives. Facebook, Twitter, Instagram s'affichent normalement dans SocialFlow.

En résumé, l'application native a accès à des mécanismes que ni les extensions Chrome ni les sites web ne peuvent utiliser. Ce n'est pas un choix marketing — c'est une contrainte technique incontournable.

Pourquoi pas Flutter ou React Native (Expo) ?

Si SocialFlow a besoin d'une application native, pourquoi ne pas utiliser Flutter ou React Native, les deux frameworks cross-platform les plus populaires ? La réponse tient en un mot : les WebViews.

Le coeur de SocialFlow, c'est d'afficher des sites web

SocialFlow n'est pas une application classique avec ses propres écrans et ses propres données. Son rôle principal est d'afficher de vrais sites web (Facebook, Twitter, Instagram...) à l'intérieur de l'application, tout en contrôlant finement ce qui se passe dans ces pages : cookies, scripts injectés, sessions isolées.

Cela signifie que le composant le plus critique de l'application est la WebView — le mini-navigateur intégré qui affiche ces sites. Et c'est là que Flutter et React Native posent problème.

Flutter : un moteur de rendu qui tourne le dos au web

Flutter dessine chaque pixel de l'interface avec son propre moteur de rendu graphique (Skia, puis Impeller). C'est ce qui lui donne des animations fluides et un rendu identique sur toutes les plateformes. Mais cette approche a un coût : Flutter ne partage rien avec le navigateur web du système.

Quand une application Flutter a besoin d'afficher une page web, elle doit intégrer un composant WebView séparé, qui fonctionne comme un corps étranger dans l'application. Les interactions entre Flutter et cette WebView sont limitées :

  • Pas d'injection de scripts au démarrage de la page. Flutter peut envoyer du JavaScript à la WebView, mais seulement après que la page a commencé à se charger. Pour SocialFlow, c'est trop tard — le mode niveaux de gris et le filtre d'amis doivent être actifs avant que le premier élément ne s'affiche, sinon l'utilisateur voit un flash de contenu non filtré.
  • Contrôle limité des cookies. Le plugin WebView de Flutter offre des fonctions basiques de gestion des cookies, mais pas le niveau de contrôle granulaire nécessaire pour sauvegarder et restaurer l'intégralité d'une session par profil, par réseau, de manière fiable.
  • Réécriture complète de l'interface. Flutter utilise Dart, un langage complètement différent. Toute l'interface de SocialFlow (écrite en Vue.js) devrait être réécrite de zéro. Cela signifie maintenir deux versions de l'interface en parallèle, avec le risque constant que les fonctionnalités divergent.

React Native (Expo) : plus proche du web, mais pas assez

React Native est plus proche du monde web que Flutter — il utilise JavaScript et React. Mais les similitudes s'arrêtent là.

  • Le composant WebView de React Native a les mêmes limitations. Le package react-native-webview ne supporte pas l'injection de scripts au tout début du chargement (avant le premier rendu). Il ne permet pas non plus de contrôler les cookies avec la granularité nécessaire.
  • Réécriture nécessaire. Même si React Native utilise JavaScript, il utilise React — pas Vue.js. L'interface de SocialFlow devrait être entièrement réécrite avec des composants React Native, qui n'ont rien en commun avec les composants Vue. Boutons, listes, animations, navigation : tout est différent.
  • Pas de version desktop convenable. React Native est conçu pour le mobile. Pour le desktop, il existe des solutions expérimentales, mais aucune n'est aussi mature que ce que propose Tauri nativement.

Ce que Tauri fait différemment

Tauri prend l'approche inverse : au lieu d'embarquer un moteur de rendu séparé, il utilise la WebView native du système d'exploitation. Sur Android, c'est la même WebView que Chrome. Sur Windows, c'est celle d'Edge. Sur macOS, celle de Safari.

Cette approche a trois conséquences majeures pour SocialFlow :

  • L'interface Vue.js fonctionne telle quelle. Pas de réécriture, pas de double maintenance. Le même code qui fait tourner l'interface est celui qui s'exécute dans l'application.
  • Les WebViews des réseaux sociaux sont natives. Tauri donne un accès direct aux API de la WebView du système, ce qui permet l'injection de scripts avant le premier rendu, le contrôle total des cookies, et la gestion des sessions isolées par profil.
  • L'application est légère. Tauri n'embarque pas de moteur de rendu — il utilise celui qui est déjà installé sur votre appareil. Résultat : l'application pèse environ 5 Mo, contre 15 Mo pour Flutter et 30 Mo ou plus pour React Native.
