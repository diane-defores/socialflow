The technical choices behind SocialFlow

No Chrome extension, no web version, no Flutter or Expo. Technical transparency is one of our values — here are the real reasons.

What SocialFlow does that's fundamentally different

SocialFlow's promise is simple: let you log into multiple accounts on each social network simultaneously, in sessions that are completely isolated from one another. Personal profile, work profile, client profile — each with its own logins, its own cookies, its own sessions.

This isn't just a dashboard that displays your networks in tabs. It's real isolation: when you're on your "Work" profile and open Facebook, your professional Facebook account appears. When you switch to the "Personal" profile, it's your personal account — without having to log out and log back in.

This isolation relies on a precise technical mechanism: each profile has its own cookie storage, completely separate from the others. This is what's known as an isolated "cookie jar."

And that's exactly where things get complicated with Chrome and the web.

Why no Chrome extension?

The fundamental problem: Chrome shares cookies across all tabs

In Chrome, all tabs share the same cookie storage. If you log into Facebook in one tab, you're logged into the same Facebook account in every other tab. This is a design choice by Google, and no extension can change this behavior.

A Chrome extension can read and modify individual cookies via Chrome's official API. But it cannot create separate cookie spaces per tab or per profile. A cookie you modify in one tab is immediately visible in all others.

The "network interception" approach: theoretically possible, practically fragile

There is a theoretical approach: intercept every outgoing network request to replace the cookies being sent, and intercept every response to capture the incoming cookies and store them in a virtual space managed by the extension. This is what some paid extensions like SessionBox do.

The problem is that Google fundamentally changed the rules with Manifest V3, the new standard for Chrome extensions:

  • Extensions can no longer modify network requests in real time. The old API that allowed this has been replaced by a declarative rules system that is far less flexible.
  • Declarative rules aren't designed for dynamic cookie swapping. They work with static or semi-static rules, not with logic that changes on every request based on the active tab.
  • Cookies set by sites in HTTP responses still land in the browser's shared storage, creating constant conflicts between profiles.

The result: even extensions that attempt this approach run into random logouts, mixed-up sessions, and incompatibilities with sites that use advanced security mechanisms (which all major social networks do).

What about Firefox?

Firefox is the only browser that offers a true isolation mechanism: Containers (Multi-Account Containers). Each container has its own cookie storage, completely separate. And Firefox extensions have access to an official API to create and manage these containers.

That's exactly what SocialFlow would need. But this API only exists in Firefox — Chrome has never implemented an equivalent, and Google has shown no intention of doing so.

We could offer a Firefox extension using Containers, but it would only serve a fraction of our users. We chose to focus our efforts on a solution that works everywhere.

What a Chrome extension could still do

A Chrome extension could offer some of SocialFlow's features: the sidebar with all your networks, grayscale mode to reduce distractions, custom script injection into pages. But without profile isolation, it would be SocialFlow stripped of its core feature. We decided that offering an incomplete product wouldn't serve anyone well.

Why no web version?

Web browsers are designed to prevent exactly what SocialFlow does

On the web, a site cannot display another site inside itself. This is a fundamental web security rule, and it exists for good reasons: it prevents a malicious site from impersonating your bank or social network by embedding it in a fake interface.

In practice, virtually all social networks (Facebook, Twitter, Instagram, LinkedIn, etc.) send an HTTP header that forbids any other site from displaying them in an embedded frame. This is a standard protection against "clickjacking" attacks.

Even if a social network didn't have this protection, the browser would prevent the SocialFlow website from accessing Facebook's or Twitter's cookies. This is the principle of the "Same-Origin Policy": each site is isolated from the others. Ironically, this is exactly the kind of isolation we'd like to control — but it's enforced by the browser, not by us.

A web version would just be a list of links

If we offered a web version, the only thing it could do would be to display links to social networks that open in new tabs. No profile isolation, no unified interface, no script injection. You might as well just use bookmarks in your browser.

That's why this site is a landing page and not the app itself.

Why the native app (Tauri) solves all of this

SocialFlow is built with Tauri, a framework that creates native applications from web technologies. Here's what that changes:

  • Each social network is displayed in a native WebView component, directly embedded in the app. It's not a browser tab: it's a mini-browser controlled by the application, with its own cookie storage.
  • The app controls the cookies of each WebView. When you switch profiles, SocialFlow saves the current profile's cookies and loads those of the new profile. Each network, in each profile, has its own cookies.
  • The app can inject scripts into WebViews from the very first moment of page loading (before the site even starts rendering). This is what enables grayscale mode, the "friends only" filter, and other customizations.
  • No display restrictions. The security headers that prevent embedding in a website don't apply to native WebViews. Facebook, Twitter, Instagram all display normally in SocialFlow.

In short, the native app has access to mechanisms that neither Chrome extensions nor websites can use. This isn't a marketing choice — it's an unavoidable technical constraint.

Why not Flutter or React Native (Expo)?

If SocialFlow needs a native app, why not use Flutter or React Native, the two most popular cross-platform frameworks? The answer comes down to one thing: WebViews.

The heart of SocialFlow is displaying websites

SocialFlow isn't a traditional app with its own screens and its own data. Its primary role is to display real websites (Facebook, Twitter, Instagram...) inside the app, while finely controlling what happens on those pages: cookies, injected scripts, isolated sessions.

This means the most critical component of the app is the WebView — the embedded mini-browser that displays these sites. And that's where Flutter and React Native fall short.

Flutter: a rendering engine that turns its back on the web

Flutter draws every pixel of the interface with its own rendering engine (Skia, then Impeller). This gives it smooth animations and identical rendering across all platforms. But this approach has a cost: Flutter shares nothing with the system's web browser.

When a Flutter app needs to display a web page, it has to integrate a separate WebView component that operates like a foreign body inside the app. The interactions between Flutter and this WebView are limited:

  • No script injection at page startup. Flutter can send JavaScript to the WebView, but only after the page has started loading. For SocialFlow, that's too late — grayscale mode and the friends filter need to be active before the first element renders, otherwise the user sees a flash of unfiltered content.
  • Limited cookie control. Flutter's WebView plugin offers basic cookie management functions, but not the granular control needed to reliably save and restore an entire session per profile, per network.
  • Complete interface rewrite. Flutter uses Dart, a completely different language. SocialFlow's entire interface (written in Vue.js) would have to be rewritten from scratch. That means maintaining two versions of the interface in parallel, with the constant risk of features diverging.

React Native (Expo): closer to the web, but not close enough

React Native is closer to the web world than Flutter — it uses JavaScript and React. But the similarities end there.

  • React Native's WebView component has the same limitations. The react-native-webview package doesn't support script injection at the very beginning of loading (before the first render). It also doesn't allow cookie control with the necessary granularity.
  • Rewrite required. Even though React Native uses JavaScript, it uses React — not Vue.js. SocialFlow's interface would have to be entirely rewritten with React Native components, which have nothing in common with Vue components. Buttons, lists, animations, navigation: everything is different.
  • No proper desktop version. React Native is designed for mobile. For desktop, there are experimental solutions, but none are as mature as what Tauri offers natively.

What Tauri does differently

Tauri takes the opposite approach: instead of bundling a separate rendering engine, it uses the operating system's native WebView. On Android, it's the same WebView as Chrome. On Windows, it's Edge's. On macOS, it's Safari's.

This approach has three major consequences for SocialFlow:

  • The Vue.js interface works as-is. No rewrite, no dual maintenance. The same code that powers the interface is the code running in the app.
  • The social network WebViews are native. Tauri gives direct access to the system WebView's APIs, enabling script injection before the first render, full cookie control, and isolated session management per profile.
  • The app is lightweight. Tauri doesn't bundle a rendering engine — it uses the one already installed on your device. Result: the app weighs about 5 MB, compared to 15 MB for Flutter and 30 MB or more for React Native.
